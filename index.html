Python for Statistics — Setup Guide

Python for Statistics

A step-by-step setup guide for R users new to Python — get JupyterLab running with a clean virtual environment in four steps.

1
Install Python
Requires admin / IT

First, check whether Python 3 is already installed:

Terminal
python3 --version

If you see a version number (e.g. Python 3.11.4), you can skip straight to Step 2. Otherwise, install Python via your distribution's package manager:

Debian / Ubuntu
sudo apt update
sudo apt install python3 python3-pip
Fedora / RHEL / CentOS
sudo dnf install python3 python3-pip
Gotcha

sudo is required only for this step. Everything from Step 2 onwards runs entirely as your regular user account — no need to involve IT again.

Tip

Most modern distributions (Ubuntu 22.04+, Fedora 38+) ship with Python 3 pre-installed. Always run python3 --version before contacting IT.

2
Install JupyterLab & run your first code
User only

Install JupyterLab into your personal user directory (no sudo needed):

Terminal
pip install --user jupyterlab

Then launch it:

jupyter lab

A browser tab opens automatically. Create a new Python 3 notebook, type the following into the first cell, and press Shift+Enter to run it:

print("Hello from Python!")
2 + 2
Gotcha — Ubuntu 23+ "externally managed environment"

On Ubuntu 23.04 and later, running pip install outside a virtual environment produces an error about an "externally managed environment". If you see this, jump ahead to Step 3 to create a virtual environment first, then come back here and install JupyterLab inside the activated venv.

Tip

If the jupyter command is not found after installing, try python3 -m jupyterlab as a workaround, or log out and back in so that ~/.local/bin is added to your PATH.

3
Create a virtual environment
User only

A virtual environment is an isolated Python installation just for your project. Think of it like a separate R library folder that cannot interfere with anything else on your system.

Terminal
python3 -m venv ~/stats-env

This creates a folder at ~/stats-env containing its own private Python interpreter and pip.

Tip

You only need to create the environment once. After that, you simply activate it at the start of each working session (Step 4).

4
Activate the venv & manage packages
User only

Activate your environment at the start of each working session:

Terminal
source ~/stats-env/bin/activate

Your shell prompt changes to show (stats-env) — you are now inside the environment and all pip commands will install only here.

Install JupyterLab inside the venv (keeps the kernel aligned with your packages):

pip install jupyterlab
jupyter lab

Install a library:

pip install pandas

Pin to a specific version:

pip install scipy==1.11.0

Upgrade to the latest version:

pip install --upgrade pandas

Install below a version ceiling (safe downgrade):

pip install "pandas<2.0"

Remove a library:

pip uninstall pandas

Freeze your environment to a file (equivalent to renv::snapshot()):

pip freeze > requirements.txt

Recreate the environment from that file on another machine:

pip install -r requirements.txt

Deactivate when you are finished:

deactivate
Gotcha — Activate every new terminal session

Opening a new terminal window drops you back to the system Python. You must run source ~/stats-env/bin/activate again each time. The activation is not persistent.

Tip

Confirm the right Python is active with which python3. If the path shown includes stats-env, you are in the correct environment.

1
Install Python
User only
Gotcha — Do not use Apple's built-in Python

macOS ships a Python stub or a stripped-down Apple-managed Python. Running python3 may trigger an Xcode installer prompt, or silently give you a version you cannot upgrade or customise. Always install Python yourself using one of the options below.

Option A — Official python.org installer (recommended for most users):

Download the macOS .pkg file from python.org and run the installer. Choose the latest stable Python 3.11 or 3.12 release.

Gotcha — python.org installer and PATH

The installer may not add Python to your shell's PATH automatically. If python3 --version still shows the Apple stub after installing, add this line to ~/.zshrc (adjust the version number to match what you installed):

export PATH="/Library/Frameworks/Python.framework/Versions/3.12/bin:$PATH"

Then apply the change with source ~/.zshrc.

Option B — Homebrew:

Terminal
# Install Homebrew if not already present (one-time)
/bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"

# Install Python
brew install python
Gotcha — Apple Silicon PATH for Homebrew

On Apple Silicon Macs (M1, M2, M3), Homebrew installs to /opt/homebrew/bin rather than /usr/local/bin (Intel). If python3 is not found after installing, add the following to your ~/.zshrc and then run source ~/.zshrc:

eval "$(/opt/homebrew/bin/brew shellenv)"
Tip

After installing, verify with python3 --version. You should see the version you installed (e.g. Python 3.12.2), not an Apple-branded stub.

2
Install JupyterLab & run your first code
User only

Install JupyterLab into your user directory:

Terminal
pip3 install --user jupyterlab

Then launch it:

jupyter lab
Gotcha — jupyter not found on PATH

On macOS, --user installs scripts to ~/Library/Python/3.x/bin, which is typically not on your PATH. If jupyter is not found, add this to ~/.zshrc (replace 3.12 with your version):

export PATH="$HOME/Library/Python/3.12/bin:$PATH"

Then run source ~/.zshrc. Alternatively, use python3 -m jupyterlab as an immediate workaround.

Gotcha — macOS Ventura+ "externally managed environment"

On macOS Ventura (13) and later, pip3 install directly may raise an "externally managed environment" error. If you see this, skip to Step 3 to create a virtual environment first, then install JupyterLab inside the activated venv.

Tip

In your first notebook, run print("Hello from Python!") in a cell with Shift+Enter. If it prints the message, everything is wired up correctly.

Gotcha — Apple Silicon and scientific libraries

On Apple Silicon (M1/M2/M3), packages like NumPy, SciPy, and Pandas require Python 3.11 or later to get native ARM64 wheels. Older Python versions fall back to Rosetta 2 (x86) builds, which are slower and may fail to install entirely.

Tip

1
Install Python
Admin for all-users install; user-only for per-user

Download the Windows installer (.exe) from python.org. Before clicking Install, read these carefully:

Critical — Tick "Add Python to PATH"

On the very first screen of the installer, make sure to check "Add python.exe to PATH" before clicking anything else. This is the single most common cause of problems for new Python users on Windows. If you missed it, re-run the installer and choose Modify.

Per-user vs. system-wide:

  • Install just for me (per-user) — the default; no admin rights required. Python installs to %LOCALAPPDATA%\Programs\Python.
  • Install for all users — requires admin rights; installs to C:\Program Files\Python3x.

For most users, the per-user install is perfectly fine and does not require IT involvement.

Gotcha — Microsoft Store Python intercept

Windows may silently redirect python and python3 commands to the Microsoft Store instead of your real installation. To disable this:

Settings → Apps → Advanced app settings → App execution aliases

Toggle off both python.exe and python3.exe entries.

Tip

After installing, open a new Command Prompt window and run python --version. You should see the version you just installed. If you see a Store prompt, revisit the App execution aliases step above.

2
Install JupyterLab & run your first code
User only

Open Command Prompt (search for "cmd" in the Start menu) and install JupyterLab:

Command Prompt
pip install jupyterlab

Then launch it:

jupyter lab

A browser tab opens automatically. Create a new Python 3 notebook, type the following, and press Shift+Enter:

print("Hello from Python!")
2 + 2
Gotcha — PowerShell execution policy

If you prefer PowerShell, JupyterLab's activation scripts may be blocked by the default execution policy. Fix this once with:

Set-ExecutionPolicy RemoteSigned -Scope CurrentUser

You only need to do this once per machine.

Gotcha — 'jupyter' not recognised

If the jupyter command is not found, Python's Scripts folder may not be on your PATH. As an immediate workaround, run JupyterLab via:

python -m jupyterlab

To fix PATH permanently, re-run the Python installer and ensure "Add Python to PATH" is ticked.

3
Create a virtual environment
User only

Create an isolated Python environment for your project. In Command Prompt, replacing YourName with your actual Windows username:

Command Prompt
python -m venv C:\Users\YourName\stats-env
Gotcha — No spaces in the path

Do not place your virtual environment in a folder whose path contains spaces (e.g. C:\My Projects\stats-env). Some tools break on paths with spaces. Keep it directly under C:\Users\YourName\.

Gotcha — Windows 260-character path limit

Windows enforces a 260-character path limit by default. If you see "path too long" errors when installing packages, ask IT to enable long paths via Group Policy. If you have admin rights yourself, you can enable it with:

reg add "HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\Control\FileSystem" /v LongPathsEnabled /t REG_DWORD /d 1 /f
Tip

You only need to create the environment once. From then on, just activate it at the start of each session (Step 4).

4
Activate the venv & manage packages
User only

Activate your environment. The command depends on which shell you are using:

Command Prompt
C:\Users\YourName\stats-env\Scripts\activate.bat
PowerShell
C:\Users\YourName\stats-env\Scripts\Activate.ps1

Your prompt changes to show (stats-env) — you are now inside the environment.

pip install jupyterlab
jupyter lab

Gotcha — "Access denied" on pip

If you see an "access denied" error when running pip install, you are most likely running pip outside a virtual environment, and pip is trying to write to a system-level folder. Always activate your venv first before installing packages.

Tip

Confirm the correct environment is active with where python. The path returned should contain stats-env\Scripts.

Tips for R users

Python's package management maps closely to R's ecosystem — the concepts just have different names.

R concept Python equivalent Notes

Switching to a project R library source ~/stats-env/bin/activate Activates the isolated environment for the current terminal session
install.packages("ggplot2") pip install matplotlib Downloads and installs a package from PyPI (Python's CRAN equivalent)
library(ggplot2) import matplotlib.pyplot as plt Loads an installed package into the current script or notebook
renv::snapshot() pip freeze > requirements.txt Records the exact version of every installed package to a file
renv::restore() pip install -r requirements.txt Recreates an environment from a previously saved snapshot

RStudio JupyterLab The interactive development environment for running and exploring code

Common gotchas

Jupyter kernel mismatch

Gotcha

If you launch JupyterLab from outside your virtual environment, the notebook's Python kernel will be the system Python — not your stats-env Python. Your installed packages will not be visible, and imports will fail. Always activate the venv first, then run jupyter lab from within it.

To check which Python a running notebook is using, run this in a cell:

import sys; print(sys.executable)

The output path should contain stats-env. If it does not, shut down Jupyter, activate the venv, and restart.
